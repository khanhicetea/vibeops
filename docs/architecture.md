# Architecture

bento separates tracked stack source, local desired state, disposable generated files, and durable runtime data. The management CLI owns the boundaries between those layers.

## Service topology

```text
Internet
   │ :80, :443/tcp, :443/udp
   ▼
Nginx (host network, /home read-only)
   │ Unix sockets under /run/php-fpm/phpXX/
   ▼
PHP-FPM phpXX ───────────────┐
                             │ Compose backend network
phpXX-runner ────────────────┼── MySQL mysqlXX:3306
  ├── Supercronic per app    └── Redis redis:6379
  └── Supervisord workers

phpXX-cli (ephemeral) runs deploy commands and shells
```

Nginx has no `ports:` mapping. It binds the host network directly. Its tracked Bento image builds pinned Zstandard filter and static modules; Zstandard is preferred, gzip is the compatibility fallback, and Brotli is not included. PHP-FPM, runners, MySQL, and Redis use the private `backend` Compose network; only shared FPM socket bind mounts connect Nginx to PHP.

For app `shop` on PHP 8.5:

```text
host:          runtime/run/php-fpm/php85/shop.sock
nginx:         /run/php-fpm/php85/shop.sock
php85:         /run/php-fpm/shop.sock
app code host: runtime/home/shop/www
containers:    /home/shop/www
```

## Ownership layers

| Layer | Paths | Owner |
|---|---|---|
| Upstream source | `config/`, `docker/`, `bento/`, `config/compose.yml` | repository |
| Local desired state | `.env`, `runtime/state/stack.json` | operator and `manage.py` |
| Generated topology/config | `compose.d/bento-*-versions.yml`, `runtime/generated/`, `runtime/secrets/` | renderer |
| Local customization | `compose.override.yml`, `compose.local.yml`, non-managed `compose.d/*.yml`, `runtime/custom/` | operator |
| Durable runtime | `runtime/home/`, backups, certs, ACME state, Docker volumes | operator |

Generated files must not be edited. App-owned custom vhost and pool templates remain under `runtime/custom/apps/<app>/`; render copies their output into the normal generated destinations.

## App model

The app slug is stack-wide unique and defines:

- private Linux user and group
- PHP-FPM pool and Unix socket
- MySQL user on exactly one managed MySQL service
- app home at `/home/<app>`
- cron and worker process identity

`runtime/state/stack.json` stores apps, a global domain index, proxies, TLS modes, managed PHP/MySQL versions, databases, cron jobs, workers, access-log settings, and custom-template selection.

An app has one main domain and zero or more aliases. All its domains share the same code tree, runtime, pool, databases, and TLS vhost. Use a separate app for a separate codebase or trust boundary.

Document-root metadata:

- empty `public_dir`: `/home/<app>/www`
- `public_dir=public`: `/home/<app>/www/public`
- `front-controller`: only `index.php` executes
- `legacy`: existing direct `.php` scripts may execute
- `auto`: front-controller for a non-empty public directory, otherwise legacy

FPM sizing is selected from the fixed `ondemand`, `balanced`, and `throughput` profiles. The image-wide `PHP_FPM_PROCESS_MAX` still caps total FPM children for a PHP version.

## Managed runtime versions

State starts with PHP 8.5 and MySQL 8.4. Rendering writes:

```text
compose.d/bento-php-versions.yml
compose.d/bento-mysql-versions.yml
```

A PHP version creates three roles:

- `phpXX`: persistent FPM
- `phpXX-runner`: persistent Supervisord process host
- `phpXX-cli`: profile-gated ephemeral shell/exec role

All roles use the same image, PHP binary, extensions, user metadata, and `/home` mount. The runner must stay at one replica.

A MySQL version creates a service such as `mysql84` and a named volume such as `mysql84-data`. MySQL version removal is intentionally not automated. Each service builds a thin Bento wrapper over its selected upstream image, adding Bash, gzip, and Zstandard for in-container backup pipelines. On ARM64, MySQL 5.7 uses `docker/mysql/5.7/Dockerfile` with `biarms/mysql:5.7` because the official 5.7 image does not provide an ARM64 variant; its compatibility entrypoint also fixes ownership of the bind-mounted log directory.

## Render and apply transaction

```bash
./manage.py render
./manage.py apply
```

Render acquires the state lock, creates a complete staging tree under `runtime/.render-txn-*`, then promotes candidate files with same-filesystem atomic replacement. Stale managed files are removed only after the candidate set is complete.

Apply adds validation and reload:

```text
lock → stage → promote → validate → reload → finalize
                   └──── rollback on validation failure
```

Validation includes the relevant running Nginx, PHP-FPM, Supervisord, and Supercronic configurations. A validation failure restores previous generated bytes. If validation succeeds but a reload signal fails, the validated new generation remains in place for a reload retry.

Most mutation commands request only the reload scope they change:

| Change | Nginx | FPM | Runner |
|---|:---:|:---:|:---:|
| app create/update | yes | yes | no |
| domain, proxy, TLS, access-log flag | yes | no | no |
| app pool customization | no | yes | no |
| cron or worker definition | no | no | matching version |
| database operation | no | no | no |
| `apply` | yes | yes | yes |

`render` never signals services. `--no-reload` allows supported mutations to be batched before a final `apply`.

## Identity and filesystem policy

Rendered user metadata creates a private UID/GID for each app. PHP-FPM, CLI commands, schedules, and workers run as that app. FPM sockets use the Nginx socket group (`SOCKET_GID`, normally `101`) so Nginx can connect without joining every private app group.

Nginx needs read/traverse access only through the selected public tree. Private credentials, SSH keys, Composer state, and logs stay private. Startup synchronizes identities but does not recursively traverse app homes.

```bash
./manage.py identity sync <app>
./manage.py permissions check <app>
./manage.py permissions fix <app> --recursive --dry-run
./manage.py permissions fix <app> --recursive
```

Recursive repair is explicit because imported trees may be large.

## Runner model

A versioned runner uses Supervisord as PID 1. Its generated programs include:

```text
runtime/generated/runner/php85/programs/bento.conf
runtime/generated/cron/php85/apps/<app>.cron
runtime/generated/cron/php85/jobs/<app>-<job>.cron
runtime/generated/cron/php85/system.cron
```

Each app with schedules gets one app-owned Supercronic child. Long-running workers are separate flat Supervisord programs. Adding one worker does not restart sibling workers or cron. A runner restart does affect every process on that PHP version.

Worker commands are stored as argv. UID/GID and filesystem permissions isolate apps, but workers in one runner share its container-level CPU and memory limits.

## Database and secret model

Each app is assigned to one MySQL service. Its user receives grants only for escaped `<app>_*` database names. Credentials are mode-600 files under:

```text
runtime/home/<app>/.credentials/<mysql-service>.env
runtime/home/<app>/.credentials/redis.env
```

Generated MySQL root option files live under `runtime/secrets/mysql/` and are mounted into only the matching MySQL container. Administrative commands avoid placing root passwords in host command arguments.

Logical backup behavior:

- stream to a private partial file
- optionally run `mysqldump | zstd -3 -T1` inside the MySQL container, streaming only compressed output to the host
- require successful, non-empty output
- atomically promote to a unique final name
- apply `--keep N` per database only after a complete successful batch
- ignore partial files and symlinks when listing/retaining

Restore streams plain SQL directly, or sends compressed bytes to an in-container `zstd -d | mysql` / `gzip -d | mysql` pipeline. It restores into either a new suffixed database or a dropped-and-recreated original and is not object-level atomic. MySQL named volumes protect against container recreation, not accidental SQL changes; off-host logical backups are still required.

## TLS and networking

Every generated vhost has HTTP and HTTPS blocks. HTTPS initially uses the tracked self-signed boot certificate. State can select NGINX ACME or external files under `runtime/certs/`.

The Nginx image and its pinned Zstandard module revision are treated as one build artifact. Dynamic compression uses the module's production defaults, `zstd_static on` serves negotiated `.zst` siblings, and `gzip_vary on` keeps shared-cache variants safe. See [nginx-zstd.md](nginx-zstd.md).

ACME state is durable and secret. HTTP-01 requires public DNS and port 80. HTTP/3 additionally requires UDP 443 through the host firewall/security group.

Because Nginx uses host networking, proxy upstream `127.0.0.1:<port>` means the host namespace. It does not resolve Compose service names on the backend bridge.

## Compose assembly

Always use `./dc` or `./manage.py compose`. The Compose service assembles:

1. `config/compose.yml`
2. generated managed-version fragments
3. `compose.override.yml`, if present
4. `compose.local.yml`, if present
5. other `compose.d/*.yml` and `compose.d/*.yaml` fragments

Inspect the merged result with `./dc config`. Compose list merge behavior can replace required mounts, so verify `/home`, generated config, sockets, logs, certificates, and ACME state after overriding service volumes.
