## Why this matters

Every app currently uses the shared `nginxsock` group as its primary process group, and startup recursively changes every app home to that group and grants group read/traverse. Because every app user belongs to the same group, this weakens app isolation. The same recursive repair runs on FPM, cron, and ephemeral CLI startup, so restart/deploy cost scales with the entire home tree and multiple containers race over the same bind mount.

This plan gives each app a private UID/GID pair, keeps the shared group only for Nginx socket/public-file access, makes container startup identity-only, and adds explicit identity/permission management commands. This is a new project: do not add migration code, compatibility aliases, or Docker-based verification.

## Required design

### Identity model

For app `shop` with state UID `10000`:

- Linux user: `shop`, UID `10000`
- Private primary group: `shop`, GID `10000`
- The app user must not be a member of `nginxsock`
- FPM worker `user` and `group`: `shop`
- FPM socket group: `nginxsock` / numeric `SOCKET_GID`

Generated `user.env` must contain:

```text
USERNAME=shop
UID=10000
GID=10000
PUBLIC_DIR=public
```

`PUBLIC_DIR` is empty for app-root document roots. Values remain unquoted because validation already restricts them to safe characters.

### Filesystem policy

`php-permissions` owns this policy and is the only recursive permission helper:

- App home `/home/<app>`: owner app, group `nginxsock`, mode `0710` (Nginx may traverse but not list/read).
- Private known directories `.ssh`, `.credentials`, `.composer`, and `logs`: owner/group `app:app`, mode `0700`; private files are not made group-readable.
- Ancestors from `/home/<app>/www` through the parent of the selected document root: owner app, group `nginxsock`, mode `0710` so Nginx can traverse.
- Selected document root (`/home/<app>/www` or `/home/<app>/www/<PUBLIC_DIR>`): owner app, group `nginxsock`.
- Under the selected document root, directories receive group read/traverse plus setgid and no `other` access; files receive group read and no `other` access. Preserve existing owner execute bits instead of normalizing every file to `0640`.
- Non-public application files remain `app:app` and inaccessible to other app users.
- Never follow symlinks outside the app tree. Use `find -P` semantics and do not dereference symlink targets.

`fix` without `--recursive` repairs only known top-level directories, ancestor traversal, and the document-root directory itself. `fix --recursive` repairs the complete private tree first, then reapplies the public-tree group/read policy. `check` may scan recursively but never mutates.

### CLI UX

Implement these commands:

```bash
./manage.py identity sync shop
./manage.py identity sync --all
./manage.py identity sync shop --php 8.5

./manage.py permissions check shop
./manage.py permissions check --all
./manage.py permissions check shop --json
./manage.py permissions fix shop
./manage.py permissions fix shop --recursive
./manage.py permissions fix shop --recursive --dry-run
```

Exactly one of positional `app_name` and `--all` is required. If `--php` is omitted for one app, use that app's state PHP version. For `--all`, group apps by their state PHP version. Unknown apps are fatal.

### Step 2: Create the identity-only container helper

Create POSIX shell script `docker/php/bin/php-identity-sync` with `set -eu`.

Requirements:

1. Accept zero usernames (sync every `*.env`) or explicit usernames.
2. Read identities from `${PHP_USERS_DIR:-/usr/local/etc/php/users.d}`.
3. Ensure the shared socket group name/GID exists. Duplicate numeric GID is allowed only for this socket group because cross-container access is numeric.
4. For each app, require safe non-empty `USERNAME`, numeric `UID`, and numeric `GID`; reject an argument whose env file does not exist.
5. Ensure a private group named exactly like the app exists at the requested GID. If the name or numeric GID conflicts, fail with an actionable message rather than using `groupadd -o`.
6. Ensure the user exists at the requested UID and private primary GID. A new user uses `useradd -u UID -g USERNAME -d /home/USERNAME -s /bin/sh -M USERNAME`.
7. For an existing user, compare `id -u` and `id -g`. If either differs, fail and instruct the operator to recreate the container; do not run `usermod -u`, recursively chown, or silently continue.
8. Ensure the app user is not a supplementary member of `nginxsock`; remove that membership if present without changing the private primary group.
9. Prepare only `${PHP_FPM_RUN_DIR:-/run/php-fpm}` as `root:nginxsock` mode `0770`.
10. Do not read or mutate app home contents. Do not rewrite `/usr/local/etc/php-fpm.d`; the Dockerfile already removes the default pool and creates `zz-vibeops.conf`.
11. Do not suppress identity/group failures with `|| true`.

Keep `PHP_USERS_DIR` and `PHP_FPM_RUN_DIR` configurable to support non-Docker tests later. Avoid `source`/`.` on env files; parse expected keys as the current helper does so generated data is not executed as shell.

**Verify**: `sh -n docker/php/bin/php-identity-sync && ! rg -n 'chown -R|find .*home|php-fpm\.d|FIX_HOME' docker/php/bin/php-identity-sync`

Expected: exit 0 and no forbidden match.

### Step 3: Create the explicit permission helper

Create `docker/php/bin/php-permissions`, a POSIX `sh` tool with this interface:

```text
php-permissions check <app> [--json]
php-permissions fix <app> [--recursive] [--dry-run] [--json]
```

It reads UID/GID/PUBLIC_DIR from `${PHP_USERS_DIR:-/usr/local/etc/php/users.d}/<app>.env`, uses `${PHP_HOME_ROOT:-/home}`, and uses `SOCKET_GROUP_NAME`/`SOCKET_GID` for Nginx-readable paths. It must call or validate `php-identity-sync <app>` before resolving names unless `--dry-run` is active; dry-run must not create users/groups.

Implementation requirements:

- Validate the app argument and env values before constructing paths.
- Reject absolute `PUBLIC_DIR`, `..` components, and any document root that does not remain under `/home/<app>/www`.
- In `check`, report each mismatch and exit `0` when clean, `1` when mismatches exist, and `2` for invalid invocation/runtime errors.
- In `fix`, print each operation in dry-run mode and otherwise apply the filesystem policy in "Required design".
- Avoid following symlinks. A symlink itself may have ownership reported, but its target must never be changed.
- Do not hide failed `chown`, `chmod`, or `find` operations. End with counts for checked, changed, and failed entries.
- `--json` emits one valid JSON object containing app, action, checked, changed, failed, and clean. Write human progress to stderr in JSON mode so stdout remains machine-readable.
- Limit all operations to one selected app; there is no implicit all-app mode inside the shell helper.

Do not attempt host-side `chown` from Python. All ownership changes remain inside the privileged PHP container invocation.

**Verify**:

```bash
sh -n docker/php/bin/php-permissions
rg -n 'check|fix|recursive|dry-run|json|PHP_HOME_ROOT|PUBLIC_DIR' docker/php/bin/php-permissions
```

Expected: shell syntax passes and all interface terms are present.

### Step 5: Wire identity-only startup and reload lifecycle

Update `docker/php/Dockerfile` to copy/chmod `php-identity-sync`, `php-permissions`, and `php-app-run`, and stop copying the deleted helper. Update both shared PHP volume lists in `compose.yml` to mount all three new helpers.

Change service lifecycle:

- FPM command: `php-identity-sync && exec php-fpm -F`
- Cron command: `php-identity-sync && exec php-supercronic`
- CLI entrypoint: `php-app-run`
- Remove `FIX_HOME_OWNERSHIP` from the shared environment.

In `vibeops/helpers.py`:

- Delete `php_disable_default_pool`; runtime FPM config rewriting no longer belongs in identity sync.
- Change `php_reload` to run `php-identity-sync <username>`, then validate/reload.
- Change `cron_reload` to run `php-identity-sync "$@"` before signaling/restarting Supercronic.
- Propagate failures; do not use `check=False` for identity sync.

In `cmd_apply` (`vibeops/runtime_commands.py`), for every running FPM service:

1. Determine app names assigned to that PHP version.
2. Execute one `php-identity-sync` call with those app names (or no names only when syncing all rendered users for that version is explicitly intended).
3. Run `php-fpm -tt`.
4. Send `USR2` only after validation succeeds.

This ordering is load-bearing: rendered pools may refer to users absent from a running container.


Expected: identity helper is present at all lifecycle points; all obsolete names are absent.

### Step 6: Add first-class identity and permission CLI handlers

Create `vibeops/permission_commands.py` and export it from `vibeops/commands.py`.

Implement shared selection logic that:

- Loads state and validates one app or `--all`.
- Uses each app's state `php_version` unless a valid explicit `--php` overrides a single app.
- Rejects `--php` with `--all` to avoid silently running apps in the wrong image.
- Groups all-app operations by PHP version.
- Requires Docker only when executing, with the existing `docker_available()` error style.

Container invocation strategy:

- For `identity sync`, prefer `docker compose exec -T <running-fpm-service> php-identity-sync <apps...>`. If FPM is stopped, run an ephemeral matching CLI service with `docker compose run --rm --entrypoint php-identity-sync <phpXX-cli> <apps...>`.
- For permission operations, prefer `docker compose exec -T <running-fpm-service> php-permissions ... <app>` one app at a time. If FPM is stopped, use the same ephemeral CLI strategy with `--entrypoint php-permissions`.
- Preserve the shell helper's exit code. In `check --all`, continue checking other apps, then exit non-zero if any app is dirty or failed.
- Forward `--recursive`, `--dry-run`, and `--json`. For multi-app JSON, Python must emit a valid JSON array, not concatenated objects; capture and parse each helper object before printing the final array.
- Never execute permission repair implicitly from `identity sync`, `apply`, startup, `exec`, or `shell`.

Register parser trees exactly as listed in "CLI UX". Use mutually exclusive required groups for positional-app versus `--all`; if argparse cannot express positional mutual exclusion cleanly, perform an explicit handler validation with an actionable error.

- The current identity source is no longer `runtime/state/stack.json` rendered into `users.d/*.env`.
- Nginx's worker group is no longer represented by `SOCKET_GROUP_NAME`/`SOCKET_GID` or sockets are no longer shared by numeric GID.

## Maintenance notes

- Adding a PHP version requires mounting all three helpers and using identity-only startup in its FPM/cron services.
- New app-private directories must not be placed under the public document root if they contain secrets.
- The socket group is an Nginx interoperability mechanism, not an app membership group. Reviewers should reject future changes that add app users to `nginxsock`.
- If automatic UID changes are needed later, design an explicit data migration command; do not put recursive ownership migration back into startup.
