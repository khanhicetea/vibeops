## Why this matters

Generated credentials use `MYSQL_*` keys, while the README tells operators to paste Laravel-style `DB_*` into app `.env`. `create_mysql_user` also **prints the password in cleartext** to stdout, which lands in shell scrollback and shared tmux sessions. Align templates with app conventions and only print the credentials **file path**.

## Current state

`config/mysql/templates/user-credentials.env.template`:

```env

MYSQL_PORT=3306
MYSQL_DATABASE_PREFIX=__USERNAME___
MYSQL_USER=__USERNAME__
MYSQL_PASSWORD=__MYSQL_PASSWORD__
```

README (site create section):

```text
DB_HOST=mysql

DB_DATABASE=myuser_app

```

`manage.py` `create_mysql_user`:

```python
    info(f"MySQL account: {username} / {password}")

```

Credentials path: `runtime/home/<user>/.credentials/mysql.env` mode `0o600`, dir `0o700`.

## Steps

### Step 1: Expand credentials template

Rewrite `config/mysql/templates/user-credentials.env.template` to include **both** conventions and comments (comments as `#` lines):

```env
# Generated by VibeOps manage.py — do not commit this file.
# Host/service name is reachable from PHP containers on the backend network.

MYSQL_HOST=mysql
MYSQL_PORT=3306
MYSQL_DATABASE_PREFIX=__USERNAME___
MYSQL_USER=__USERNAME__
MYSQL_PASSWORD=__MYSQL_PASSWORD__

# Laravel / many PHP apps
DB_HOST=mysql
DB_PORT=3306
DB_USERNAME=__USERNAME__
DB_PASSWORD=__MYSQL_PASSWORD__
# DB_DATABASE is per-site: {prefix}{db_name} e.g. __USERNAME___app
```

Keep placeholders compatible with existing `write_template` keys: `USERNAME`, `MYSQL_PASSWORD` (see `create_mysql_user` values dict). If you need `__USERNAME___` literal prefix in comments, use the same `__USERNAME__` substitution already used.

**Do not** put a fake `DB_DATABASE` unless you also pass it from create_mysql_user (user create has no site db yet). Prefix-only is correct.

### Step 2: Stop printing passwords

In `create_mysql_user`, change success messages to:

```python
    info(f"MySQL account ready: user={username}")
    info(f"Credentials saved (mode 600): vibeops/{rel(cred_path)}")
    info("Password is only in that file; not printed here.")
```

Never `info(... password ...)`.

If the password was user-supplied via `--mysql-password`, still do not echo it.

### Step 3: README alignment

Update the “PHP app connection values” section to say:

1. Generated file path pattern: `runtime/home/<user>/.credentials/mysql.env`
2. File contains both `MYSQL_*` and `DB_*`
3. `DB_DATABASE` / full name is `{user}_{db_name}` when using `site create ... <db_name>`

### Step 4: Verify template rendering keys

Confirm `write_template` / `render_template_text` replaces `__USERNAME__` and `__MYSQL_PASSWORD__` (existing convention — read `render_template_text` in manage.py).

```bash
grep -n 'render_template_text\|__.*__' manage.py | head -40
```

Match existing placeholder style exactly.

**Verify**:

```bash
python3 -m py_compile manage.py
grep -n 'DB_HOST\|DB_PASSWORD\|MYSQL_PASSWORD' config/mysql/templates/user-credentials.env.template
grep -n 'MySQL account:' manage.py || true
```

→ Template has DB_* and MYSQL_*. No `MySQL account: {username} / {password}` style line.
