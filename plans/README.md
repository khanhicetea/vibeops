## Findings addressed

- Shared primary `nginxsock` group weakens app-to-app filesystem isolation.
- FPM, cron, and ephemeral CLI startup repeatedly traverse/chown complete app homes.
- Identity creation, FPM setup, socket setup, and permission repair are coupled in one helper.
- Existing UID mismatches are silently left inconsistent until container recreation.
- `manage.py apply` validates/reloads FPM before ensuring rendered users exist.
- Operators lack supported identity sync and permission check/fix commands.
- Permission failures are widely hidden by best-effort `|| true` behavior.
- There is no automated verification baseline for this security-sensitive path.

## Findings considered and rejected

- **Persist `/etc/passwd` and `/etc/group` in Docker volumes**: rejected because it couples mutable identity databases to image internals and complicates upgrades. Cheap identity reconciliation belongs at container start.
- **Keep recursive repair at startup with a marker/hash**: rejected because multiple role containers still own filesystem policy and stale markers can hide drift. Repair is explicit instead.
- **Retain `php-user-sync` as a compatibility wrapper**: rejected because the operator said this is a new project with no migration/compatibility requirement.
- **Add migration logic for existing shared-group homes**: rejected for the same reason; regenerate/recreate development state instead.
- **Run Docker integration tests locally**: rejected by explicit operator instruction and because Docker is unavailable on the macOS development machine. Plans use import, mock, shell-syntax, dry-run, and source-invariant tests only.
