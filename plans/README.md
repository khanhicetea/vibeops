## Dependency notes

- Plan 001 lands first because every behavior change needs a discoverable local and CI verification gate.
- Plans 002 and 003 remove known state-corruption paths before Plan 004 introduces a single state transaction boundary.
- Plan 005 builds on Plan 004 so generated files and desired state share one serialized apply lifecycle.
- Plan 006 should land before Plans 008 and 010 so new database subprocesses do not add more bare `docker compose` calls.
- Plan 007 follows Plans 005 and 006 because the skill/runbook must document the final render/apply and Compose behavior, not an intermediate interface.
- Plan 009 follows Plan 002 because FPM profile resolution is app metadata and must not reintroduce implicit PHP-version changes.
- Plan 011 is intentionally last. It is a behavior-preserving structural refactor and should consume the expanded tests from all earlier plans.

## Direction items intentionally deferred

These are not rejected, but they need separate design/spike plans after the reliability work above:

- A local declarative version catalog that can generate Compose fragments for additional PHP/MySQL versions without editing upstream-owned `compose.yml`.
- First-class per-app Nginx cache, streaming, body-limit, and custom-include settings.
- A `doctor`, `render --check`, and `render --diff` update-safety UX.
- Automated image/dependency update PRs after Linux smoke CI is stable.

## Findings considered and rejected

- Replace host-network Nginx or Unix PHP-FPM sockets: rejected; these are explicit project architecture choices and no evidence showed they are the current bottleneck.
- Enable proxy/FastCGI caching globally: rejected; framework/session correctness requires opt-in policy rather than a blanket performance switch.
- Treat local proxy upstream text as a remote injection boundary: rejected as a security finding because the CLI input comes from an already privileged local operator. Input validation remains a worthwhile future correctness improvement.
- Replace the no-dependency Python CLI with a framework: rejected; this would increase bootstrap and upstream maintenance cost without addressing the concrete findings.
